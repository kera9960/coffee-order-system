# 주문 정책

## 규칙

### 주문 구성

- 한 주문에는 여러 메뉴를 포함할 수 있다.
- 주문에는 최소 하나 이상의 메뉴가 포함되어야 한다.
- 한 주문에 하나의 메뉴만 포함하는 단일 메뉴 주문도 허용한다.
- 한 주문에서 같은 메뉴는 하나의 주문 항목으로 표현한다.
- 같은 메뉴를 여러 개 주문할 때는 수량으로 표현한다.
- 같은 주문 요청에 동일한 메뉴 ID를 여러 항목으로 중복해서 전달할 수 없다.
- 각 주문 항목의 수량은 1 이상이어야 한다.

### 주문 금액

- 주문 요청에는 메뉴 ID와 수량만 전달한다.
- 메뉴 가격과 주문 총액은 클라이언트가 결정하지 않는다.
- 서버가 메뉴 가격과 주문 수량을 기준으로 각 주문 항목의 금액을 계산한다.
- 주문 총액은 모든 주문 항목 금액의 합으로 계산한다.

### 주문 시점 정보 보존

- 주문 항목에는 주문 당시의 메뉴 이름과 가격을 보존한다.
- 주문 완료 이후 메뉴 이름이나 가격이 변경되어도 과거 주문 내역은 변경하지 않는다.
- 과거 주문 조회에서는 현재 메뉴 정보가 아니라 주문 당시 저장한 정보를 사용한다.

### 메뉴 판매 상태

- 신규 주문에는 판매 중인 메뉴만 포함할 수 있다.
- 메뉴 판매 가능 여부는 `docs/policy/menu-sales.md`를 따른다.

### 포인트 결제

- 주문은 포인트로만 결제한다.
- 포인트 결제 규칙은 `docs/policy/point.md`를 따른다.
- 고객의 포인트가 주문 금액보다 부족하면 주문할 수 없다.

### 주문 완료

- 주문 정보, 주문 항목, 포인트 차감, 포인트 사용 이력이 모두 정상적으로 처리되어야 주문이 완료된다.
- 처리 과정 중 하나라도 실패하면 주문은 완료되지 않아야 한다.
- 실패한 주문으로 인해 포인트만 차감되거나 주문 데이터만 저장되어서는 안 된다.

### 주문 조회

- 고객은 고객 ID에 해당하는 주문 목록을 조회할 수 있다.
- 고객은 고객 ID와 주문 ID에 해당하는 주문 상세를 조회할 수 있다.
- 현재 프로젝트에는 인증 기능이 없으므로 고객 식별은 요청에 전달된 고객 ID를 기준으로 한다.

### 현재 범위

- 현재 범위에는 주문 취소 기능을 포함하지 않는다.
- 현재 범위에는 취소 주문의 포인트 환불 기능을 포함하지 않는다.
- 주문 취소와 환불 기능이 추가되면 별도의 정책을 정의한다.

## 근거 / 배경

주문 금액을 클라이언트가 전달하면 실제 메뉴 가격과 다른 값으로
주문을 시도할 수 있으므로 서버가 주문 금액을 계산한다.

메뉴 정보가 변경된 이후에도 사용자가 주문 당시 정보를 확인할 수 있도록
메뉴 이름과 가격을 주문 항목에 보존한다.

주문과 포인트 결제는 하나의 비즈니스 작업이므로
일부 데이터만 반영되는 상태를 허용하지 않는다.

## 적용 대상

- 주문 생성
- 주문 목록 조회
- 주문 상세 조회
- 포인트 결제
- 포인트 사용 이력 기록
