# Design Documents

## 목적

이 디렉터리는 비즈니스 정책을 코드로 구현하는 방법을 기능별로 정의한다.

설계 문서는 다음 내용을 다룬다.

- 패키지와 주요 클래스
- 계층별 책임
- 처리 순서
- 검증 위치
- 트랜잭션 범위
- 조회와 저장 방식
- 주요 예외
- 테스트 포인트
- 구현 방식 선택 이유

## 문서 책임

각 문서는 다음 책임을 가진다.

```text
docs/policy/          비즈니스 규칙
docs/api/README.md    외부 API 계약
docs/db/README.md     데이터 구조와 제약조건
docs/design/          구체적인 구현 방법
```

설계 문서에서 정책, API와 DB 내용을 그대로 반복하지 않는다.

구현 기준이 필요한 경우 관련 문서를 링크하여 참조한다.

## 문서 구조

```text
docs/
└── design/
    ├── README.md
    ├── menu/
    │   └── design.md
    ├── point/
    │   └── design.md
    ├── order/
    │   └── design.md
    └── popular-menu/
        └── design.md
```

간단한 기능은 별도의 설계 문서를 만들지 않을 수 있다.

다음과 같은 경우 기능별 설계 문서를 작성한다.

- 여러 도메인이 함께 사용됨
- 트랜잭션 처리가 필요함
- 복잡한 조회나 집계가 필요함
- 구현 선택지가 여러 개임
- 주요 설계 결정의 이유를 기록할 필요가 있음

## 작성 원칙

- 비즈니스 규칙은 `docs/policy/`를 기준으로 한다.
- API 요청과 응답은 `docs/api/README.md`를 기준으로 한다.
- 테이블과 제약조건은 `docs/db/README.md`를 기준으로 한다.
- 현재 프로젝트 구조와 코드 패턴을 우선한다.
- 구현되지 않은 내용을 완료된 것처럼 작성하지 않는다.
- 정책에 없는 새로운 규칙을 설계 문서에서 만들지 않는다.
- 구현 방식이 변경되면 관련 설계 문서를 함께 수정한다.

## 기본 구성

기능별 설계 문서는 필요한 항목만 선택하여 작성한다.

```text
## 목적
## 관련 문서
## 구현 범위
## 패키지와 주요 클래스
## 처리 흐름
## 검증 책임
## 트랜잭션
## 조회와 저장 방식
## 예외 처리
## 테스트 포인트
## 설계 결정
```

모든 항목을 형식적으로 채우지 않는다.

기능이 단순하면 처리 흐름과 주요 클래스만 작성할 수 있다.
