# Plan Guide

## 목적

Plan은 사용자의 요청과 현재 프로젝트를 분석하고,
구현 방법과 작업 범위를 확정하는 단계다.

Plan에서는 파일을 변경하지 않는다.

## 시작 전 확인

요청과 관련된 다음 내용을 확인한다.

- 현재 프로젝트 구조
- 관련 코드와 테스트
- 기존의 유사한 구현
- 관련 정책, API와 DB 문서
- 기능 설계 문서
- 현재 Git 변경 상태

문서 위치는 루트 `AGENTS.md`의 컨텍스트 맵을 따른다.

파일을 읽지 않고 프로젝트 구조나 기존 구현을 추측하지 않는다.

## Plan 작성 원칙

Plan에는 다음 내용을 포함한다.

- 작업 목적
- 현재 구현 상태
- 구현 선택지와 장단점
- 추천안과 추천 이유
- 예상 변경 영역
- 정책, API와 DB 영향
- 작업 순서
- 테스트 계획
- 완료 조건
- 위험 요소와 확인 사항

선택지가 여러 개라면 현실적인 방법을 비교한다.

선택지가 사실상 하나라면
불필요하게 여러 안을 만들지 않고 그 이유를 설명한다.

확인하지 못한 내용은 이미 존재하는 것처럼 작성하지 않는다.

## 정책과 문서

- 비즈니스 규칙은 `docs/policy/`를 기준으로 한다.
- 정책에 없는 내용을 임의로 확정하지 않는다.
- 정책과 요청이 충돌하면 Plan에서 보고한다.
- 정책 변경 시 영향을 받는 API, DB, 코드와 테스트를 함께 검토한다.
- API 또는 DB 변경이 필요하면 변경 내용을 Plan에 명시한다.
- 위험 작업은 `docs/safety-guide.md`를 따른다.

## 테스트 계획

최소한 다음 항목을 검토한다.

- 정상 동작
- 주요 실패 조건
- 비즈니스 정책 위반
- 경계값
- 데이터 일관성
- 관련 기존 기능 영향

각 테스트가 무엇을 검증하는지 설명한다.

## 완료 조건

작업 완료 여부를 확인할 수 있는 기준을 작성한다.

예시:

- 요청한 기능이 정책에 맞게 동작한다.
- 실패 시 일부 데이터만 저장되지 않는다.
- 관련 테스트가 통과한다.
- API와 DB 문서가 구현과 일치한다.

## 결과 형식

```text
## 작업 목적

## 현재 상태

## 구현 선택지

## 추천안

## 예상 변경 영역

## 정책·API·DB 영향

## 작업 순서

## 테스트 계획

## 완료 조건

## 위험 요소 및 확인 사항
```

## 사용자 승인

Plan을 제시한 뒤 사용자의 명시적인 승인을 기다린다.

승인 예시:

- `승인`
- `진행해줘`
- `추천안으로 진행해줘`
- `A안으로 진행해줘`

질문, 설명 요청과 계획 수정 요청은 승인으로 보지 않는다.

승인이 불명확하면 Generate 단계로 이동하지 않는다.

## 허용되는 작업

프로젝트를 이해하기 위한 읽기 작업만 수행한다.

- 코드와 문서 확인
- 테스트 확인
- 파일과 디렉터리 조회
- `git status`, `git diff`, `git log` 확인

## 금지 사항

- 코드, 테스트, 문서와 설정 변경
- 파일 생성, 이동 또는 삭제
- DB와 의존성 변경
- 자동 포맷 적용
- commit, push, merge

실제 변경은 사용자 승인 후 Generate 단계에서만 수행한다.
