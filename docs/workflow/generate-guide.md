# Generate Guide

## 목적

Generate는 사용자가 승인한 Plan을 코드와 파일에 반영하는 단계다.

사용자가 승인한 구현 방식과 변경 범위만 작업한다.

## 진입 조건

다음 조건을 모두 만족해야 한다.

- Plan이 작성되었다.
- 구현 방식과 변경 범위가 확정되었다.
- 테스트 계획과 완료 조건이 정리되었다.
- 사용자가 명시적으로 승인했다.
- 위험 작업이 있다면 해당 작업도 승인받았다.

승인이 불명확하면 파일을 변경하지 않는다.

## 시작 전 확인

다음 내용을 확인한 뒤 구현한다.

- 승인된 Plan
- 관련 코드와 테스트
- 관련 비즈니스 정책
- API와 DB 문서
- 코드 작성 규칙
- 현재 Git 변경 상태
- 사용자가 이미 수정한 파일

문서 위치는 루트 `AGENTS.md`의 컨텍스트 맵을 따른다.

## 구현 원칙

- 승인된 범위만 변경한다.
- 기존 프로젝트 구조와 코드 패턴을 우선한다.
- 요구사항을 충족하는 최소 범위만 수정한다.
- 관련 없는 리팩터링, 이름 변경, 파일 이동과 포맷 변경을 하지 않는다.
- 사용자의 기존 변경사항을 훼손하거나 되돌리지 않는다.
- 읽지 않은 파일의 구조를 추측하여 수정하지 않는다.
- 테스트를 통과시키기 위해 요구사항이나 검증을 약화하지 않는다.
- 임시 코드와 사용하지 않는 코드를 남기지 않는다.

## 정책과 문서

- 비즈니스 규칙은 `docs/policy/`를 따른다.
- API 계약은 `docs/api/README.md`를 따른다.
- DB 구조는 `docs/db/README.md`를 따른다.
- 코드 작성 방식은 `docs/code-convention.md`를 따른다.
- 구체적인 구현 방법은 해당 기능의 `design.md`에 기록한다.
- 정책 내용을 API, DB와 기능 설계 문서에 중복해서 정의하지 않는다.

정책, API 계약 또는 DB 구조를 변경해야 한다면
승인된 Plan에 포함되어 있는지 확인한다.

## 테스트

Plan의 테스트 계획과 완료 조건을 기준으로 테스트를 작성하거나 수정한다.

최소한 다음 항목을 검토한다.

- 정상 동작
- 주요 실패 조건
- 비즈니스 정책 위반
- 경계값
- 데이터 일관성
- 관련 기존 기능 영향

다음 방식으로 테스트 실패를 숨기지 않는다.

- 테스트 삭제 또는 비활성화
- 검증 구문 제거
- 요구사항과 다른 기대값으로 변경
- 예외를 무조건 무시하는 처리

## 작업 순서

1. 승인된 Plan과 완료 조건 확인
2. 관련 코드와 테스트 확인
3. 필요한 테스트 작성 또는 수정
4. 승인된 범위 구현
5. 관련 문서 갱신
6. 변경 파일과 승인 범위 확인
7. Evaluate 단계로 이동

## Plan으로 돌아가는 조건

다음 상황에서는 작업을 중단하고 Plan으로 돌아간다.

- 기존 정책과 충돌함
- 새로운 공통 정책이 필요함
- 승인되지 않은 API 또는 DB 변경이 필요함
- 새로운 의존성이나 설정 변경이 필요함
- 보안 구조 변경이 필요함
- 파일, API 또는 기능 삭제가 필요함
- 예상보다 큰 리팩터링이 필요함
- 사용자의 기존 변경사항과 충돌함
- 승인된 방법으로 요구사항을 충족하기 어려움
- 작업 범위가 크게 확대됨

Plan으로 돌아갈 때 다음 내용을 보고한다.

- 새로 확인한 사실
- 기존 Plan으로 진행할 수 없는 이유
- 추가로 필요한 변경
- 영향 범위
- 가능한 선택지

## 종료 조건

다음 조건을 만족하면 Evaluate 단계로 이동한다.

- 승인된 구현을 완료했다.
- 필요한 테스트를 작성하거나 수정했다.
- 관련 문서를 갱신했다.
- 승인되지 않은 변경이 없다.
- 사용자의 기존 변경사항을 보존했다.
- 임시 코드나 불필요한 변경이 남아 있지 않다.

Generate 단계에서는 최종 성공을 선언하지 않는다.
컴파일, 테스트, 빌드와 요구사항 충족 여부는 Evaluate에서 검증한다.

## 금지 사항

- 승인 범위를 벗어난 변경
- 정책의 임의 변경
- 관련 없는 리팩터링
- 테스트 삭제 또는 비활성화
- 사용자 변경사항 되돌리기
- 비밀정보 작성
- 운영 데이터나 운영 환경을 사용한 테스트
- commit, push, merge

위험 작업과 Git 작업의 세부 규칙은 다음 문서를 따른다.

- `docs/safety-guide.md`
- `docs/git-guide.md`
