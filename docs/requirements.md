# Requirements

## 문서 목적

이 문서는 원본 과제에서 요구한 필수 기능과 현재 프로젝트에서 확장한 요구사항을 구분하여 정의한다.

요구사항 문서는 무엇을 만들어야 하는지를 다룬다.
구현 방법, 패키지 구조, 트랜잭션 처리 방식, 조회 쿼리와 같은 기술 설계는
정책 문서와 기능별 `design.md`에서 다룬다.

## 원본 필수 요구사항

### 1. 커피 메뉴 목록 조회

- 커피 메뉴 정보를 조회할 수 있어야 한다.
- 응답에는 메뉴 ID, 메뉴 이름, 가격이 포함되어야 한다.

### 2. 포인트 충전

- 결제 수단은 포인트만 사용한다.
- 사용자 식별값과 충전 금액을 입력받아 포인트를 충전할 수 있어야 한다.
- 1원은 1포인트로 계산한다.

### 3. 커피 주문 및 결제

- 사용자 식별값과 메뉴 ID를 입력받아 주문과 결제를 처리할 수 있어야 한다.
- 결제는 포인트만 사용한다.
- 주문 금액만큼 충전된 포인트에서 차감해야 한다.
- 초기 데이터, Mock API 또는 테스트 코드로 사용자 식별값, 메뉴 ID와 결제 금액을 전달할 수 있어야 한다.

### 4. 인기 메뉴 목록 조회

- 최근 7일 동안의 주문을 기준으로 인기 메뉴 3개를 조회할 수 있어야 한다.
- 메뉴별 주문 횟수를 정확하게 집계해야 한다.

### 5. 도전 요구사항

- 애플리케이션이 여러 서버와 여러 인스턴스로 실행되어도 기능이 올바르게 동작해야 한다.
- 동시성 문제를 고려해야 한다.
- 데이터 일관성을 고려해야 한다.
- 각 기능과 예외 상황에 대한 테스트를 작성해야 한다.

## 프로젝트 확장 요구사항

### 고객

- 고객 정보를 조회할 수 있어야 한다.
- 고객별 현재 포인트 잔액을 조회할 수 있어야 한다.
- 고객별 포인트 충전과 사용 이력을 조회할 수 있어야 한다.

### 메뉴 관리

- 메뉴를 등록할 수 있어야 한다.
- 메뉴 상세 정보를 조회할 수 있어야 한다.
- 메뉴 이름과 가격을 수정할 수 있어야 한다.
- 메뉴 판매 상태를 변경할 수 있어야 한다.
- 판매 중지된 메뉴는 일반 메뉴 목록과 신규 주문에서 제외할 수 있어야 한다.

### 주문

- 한 주문에 여러 메뉴를 포함할 수 있어야 한다.
- 같은 메뉴를 여러 개 주문할 때 수량으로 표현할 수 있어야 한다.
- 여러 메뉴 주문 기능은 단일 메뉴 주문을 막아서는 안 된다.
- 주문 당시 메뉴 이름과 가격을 보존해야 한다.
- 고객별 주문 목록과 주문 상세를 조회할 수 있어야 한다.

### 인기 메뉴

- 판매 중인 메뉴만 인기 메뉴 결과에 포함할 수 있어야 한다.
- 인기 메뉴 정렬 기준이 같은 경우 일관된 순서로 반환할 수 있어야 한다.

## 범위에서 제외하는 기능

- 회원가입과 로그인
- 인증과 인가
- 현금, 카드와 외부 결제
- 주문 취소와 환불
- 재고 관리
- 쿠폰과 할인
- 매장과 지점 관리
- 배송과 포장 상태 관리
- 실제 결제 또는 외부 시스템 연동
