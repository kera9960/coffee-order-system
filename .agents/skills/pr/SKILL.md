# PR 작성 스킬

## 목적

현재 작업에서 실제로 변경된 내용을 분석하여 Pull Request 제목과 본문 초안을 작성한다.

정해진 PR 템플릿의 빈칸을 채우는 방식으로 작성하지 않는다.

변경 목적, 구현 흐름, 설계 판단, 검증 결과를 바탕으로 리뷰어가 변경사항을 빠르게 이해할 수 있는 구조를 스스로 구성한다.

이 스킬은 PR 문서 작성만 담당한다.

코드 수정, 커밋, 푸시, 브랜치 병합, 실제 Pull Request 생성은 수행하지 않는다.

---

## 기본 원칙

* 저장소에서 직접 확인한 사실만 작성한다.
* 대화 내용만 믿지 않고 실제 변경사항을 확인한다.
* 파일 목록보다 변경된 기능과 동작을 중심으로 설명한다.
* 실행하지 않은 테스트를 통과했다고 작성하지 않는다.
* 확인되지 않은 성능 개선이나 효과를 주장하지 않는다.
* 관련 없는 변경사항을 하나의 PR에 임의로 포함하지 않는다.
* 사용자가 작성한 코드를 에이전트가 작성한 것처럼 표현하지 않는다.
* 리뷰어가 알아야 할 내용이 없다면 불필요한 섹션을 만들지 않는다.
* 모든 PR에 동일한 제목과 섹션 구조를 강제하지 않는다.

---

## 확인해야 할 자료

PR을 작성하기 전에 다음 자료를 확인한다.

1. 루트의 `AGENTS.md`
2. 승인된 작업 계획
3. 이번 작업에서 구현한 내용
4. Evaluate 단계의 테스트 및 검증 결과
5. 현재 브랜치의 실제 Git 변경사항
6. 변경과 관련된 요구사항, 정책, 설계 문서
7. 저장소에 기존 Pull Request 작성 규칙이 있다면 해당 규칙

문서가 없거나 확인할 수 없는 경우 존재한다고 가정하지 않는다.

---

## 변경사항 확인

저장소 규칙에서 허용하는 읽기 전용 Git 명령을 사용하여 현재 상태를 확인한다.

예시:

```bash
git status --short
git branch --show-current
git diff --stat
git diff
git diff --staged
git log --oneline
```

기준 브랜치가 명확한 경우에는 기준 브랜치와 현재 브랜치의 차이도 확인한다.

기준 브랜치는 사용자가 지정하거나 저장소 규칙에서 확인된 경우에만 사용한다.

`main` 또는 `develop`을 임의로 기준 브랜치로 선택하지 않는다.

추적되지 않은 파일은 `git diff`에 표시되지 않을 수 있으므로 `git status --short`에서 별도로 확인한다.

---

## 작업 범위 판단

변경사항을 확인한 뒤 다음 내용을 구분한다.

* 이번 작업에서 발생한 변경
* 이전부터 존재하던 변경
* 현재 작업과 관련 없는 변경
* 작업 목적과의 관련성이 불분명한 변경

관련성이 불분명한 변경은 PR에 포함된 것으로 단정하지 않는다.

서로 다른 목적의 변경이 섞여 있다면 하나의 작업처럼 묶지 않는다.

필요한 경우 다음과 같이 알려준다.

```text
현재 변경사항에 서로 다른 작업이 섞여 있어 하나의 PR로 설명하기 어렵습니다.
PR 범위를 나누거나 포함할 변경사항을 확인해야 합니다.
```

---

## 변경사항 분석 기준

단순히 수정된 파일을 나열하지 않는다.

다음 관점에서 변경사항을 분석한다.

### 변경 목적

* 어떤 문제를 해결하려는 작업인지
* 어떤 요구사항을 구현한 것인지
* 기존 동작에서 무엇이 부족했는지

### 동작 변화

* 사용자가 경험하는 동작이 어떻게 달라졌는지
* 요청과 응답의 흐름이 어떻게 바뀌었는지
* 데이터가 어떤 순서로 저장되거나 조회되는지
* 예외 상황에서 어떤 동작을 하는지

### 설계 판단

* 왜 현재 방식으로 구현했는지
* 중요한 책임을 어느 객체 또는 계층에 배치했는지
* 트랜잭션 경계를 어떻게 설정했는지
* 데이터 정합성을 어떻게 보장하는지
* 다른 대안과 비교해 어떤 선택을 했는지

설계 판단은 코드나 문서에서 근거를 확인할 수 있을 때만 작성한다.

### 영향 범위

* 변경된 API
* 변경된 데이터 구조
* 기존 기능에 미치는 영향
* 마이그레이션 또는 호환성 문제
* 후속 작업이 필요한 부분

관련 있는 내용만 포함한다.

### 검증 결과

* 실행한 테스트
* 통과한 테스트
* 실패한 테스트
* 실행하지 못한 테스트
* 수동으로 확인한 동작
* 확인하지 못한 위험 요소

---

## PR 제목 작성

PR 제목은 가장 중요한 변경 목적을 한 줄로 표현한다.

저장소에 제목 규칙이 있으면 해당 규칙을 우선한다.

별도의 규칙이 없다면 다음 형식을 참고한다.

```text
<type>: <핵심 변경 내용>
```

사용 가능한 type 예시:

* `feat`: 새로운 기능
* `fix`: 오류 수정
* `refactor`: 외부 동작을 유지한 구조 개선
* `test`: 테스트 추가 또는 수정
* `docs`: 문서 변경
* `chore`: 설정, 빌드 또는 기타 작업

제목 작성 기준:

* 구현 파일 이름보다 기능을 표현한다.
* 제목만 읽어도 변경 목적을 알 수 있게 한다.
* 너무 많은 내용을 한 문장에 넣지 않는다.
* 마침표를 붙이지 않는다.
* 확인되지 않은 결과를 표현하지 않는다.
* 기본적으로 가장 적절한 제목 하나만 제안한다.

여러 제목이 모두 타당하고 사용자의 선택이 필요한 경우에만 대안을 추가한다.

---

## PR 본문 구성

고정된 PR 템플릿을 사용하지 않는다.

변경사항의 성격에 따라 필요한 내용을 선택하여 자연스러운 순서로 구성한다.

일반적으로 다음 흐름을 참고할 수 있다.

1. 변경 배경과 목적
2. 구현된 핵심 동작
3. 중요한 설계 판단
4. 데이터 또는 요청 처리 흐름
5. 검증 결과
6. 리뷰어가 확인해야 할 부분
7. 제한사항 또는 후속 작업

모든 항목을 반드시 포함할 필요는 없다.

변경사항이 작다면 짧게 작성한다.

변경사항이 복잡하다면 기능별로 구분하여 작성한다.

---

## 변경 유형별 작성 기준

### 기능 추가

다음을 중심으로 작성한다.

* 사용자가 새롭게 할 수 있는 일
* 주요 처리 흐름
* 적용된 정책
* 정상 및 예외 동작
* 기존 기능과의 연결 관계

### 버그 수정

다음을 중심으로 작성한다.

* 기존에 발생하던 문제
* 문제의 원인
* 수정한 방식
* 재발 방지를 위한 테스트
* 다른 기능에 미치는 영향

### 리팩터링

다음을 중심으로 작성한다.

* 기존 구조의 문제
* 변경된 책임과 의존 관계
* 외부 동작이 유지되는지
* 구조 변경으로 얻는 이점
* 검증 방법

단순히 “코드를 정리했다”라고 작성하지 않는다.

### 데이터베이스 변경

다음을 중심으로 작성한다.

* 테이블, 컬럼, 제약조건 또는 인덱스 변경
* 데이터 정합성에 미치는 영향
* 기존 데이터와의 호환성
* 마이그레이션 필요 여부
* 조회 또는 저장 흐름의 변화

### API 변경

다음을 중심으로 작성한다.

* 추가되거나 변경된 API
* 요청 및 응답의 핵심 변화
* 유효성 검증
* 예외 응답
* 기존 클라이언트에 미치는 영향

### 동시성 또는 트랜잭션 변경

다음을 중심으로 작성한다.

* 동시에 요청이 들어올 때 발생할 수 있는 문제
* 사용한 제어 방식
* 락 또는 트랜잭션의 범위
* 실패 및 롤백 처리
* 테스트한 경쟁 상황

---

## 검증 결과 작성

검증 결과는 실제 실행 기록을 기준으로 작성한다.

성공한 검증과 확인하지 못한 검증을 구분한다.

예시:

```markdown
## 검증

- 포인트 충전 서비스 단위 테스트 통과
- 주문 생성 통합 테스트 통과
- 전체 Gradle 테스트 통과

확인하지 못한 항목:

- 실제 MySQL 환경의 동시 요청 테스트는 실행하지 않음
```

체크박스를 사용해야 할 이유가 없다면 일반 목록으로 작성한다.

실행하지 않은 검증을 빈 체크박스로 나열하는 방식도 강제하지 않는다.

테스트가 실패했다면 실패 사실과 현재 상태를 숨기지 않는다.

---

## 리뷰 포인트 작성

리뷰 포인트는 변경 파일 목록이 아니다.

리뷰어의 판단이 필요한 내용만 작성한다.

예시:

* 포인트 차감과 주문 저장을 하나의 트랜잭션으로 묶은 범위가 적절한지
* 판매 중지 메뉴 검증을 주문 서비스에서 수행하는 책임 배치가 적절한지
* 인기 메뉴 집계의 동률 정렬 기준이 정책과 일치하는지
* 데이터베이스 제약조건과 애플리케이션 검증의 역할 분리가 적절한지

특별히 검토가 필요한 내용이 없다면 리뷰 포인트 섹션을 생략한다.

---

## 작성하지 말아야 할 표현

다음과 같은 모호한 표현은 피한다.

```text
코드를 개선했습니다.
로직을 수정했습니다.
성능을 최적화했습니다.
예외 처리를 추가했습니다.
테스트를 완료했습니다.
```

무엇이 어떻게 달라졌는지 구체적으로 작성한다.

예시:

```text
주문 생성 전에 각 메뉴의 판매 상태를 확인하고,
판매 중지된 메뉴가 포함된 경우 주문 저장과 포인트 차감을 수행하지 않도록 변경했습니다.
```

## 출력 형식

최종 결과는 사용자가 GitHub에 바로 복사할 수 있도록 Markdown으로 작성한다.

기본 출력 형식:

```markdown
## 추천 PR 제목

feat: 주문 생성 및 포인트 차감 기능 추가

## PR 본문

PR 본문 내용
```

범위가 불분명하거나 검증 결과가 부족한 경우에만 마지막에 별도로 표시한다.

```markdown
## 확인이 필요한 부분

- 현재 변경사항에 메뉴 관리와 주문 기능 변경이 함께 포함되어 있습니다.
- 두 기능을 하나의 PR로 제출할지 확인이 필요합니다.
```

PR 본문 안에 에이전트의 분석 과정이나 Git 명령 실행 과정을 작성하지 않는다.

---

## 금지 사항

이 스킬을 사용할 때 다음 작업을 수행하지 않는다.

* 저장소 파일 생성 또는 수정
* 코드 구현 또는 리팩터링
* 테스트 코드 작성
* 커밋 생성
* 원격 저장소 푸시
* 브랜치 생성 또는 삭제
* 브랜치 병합 또는 rebase
* `gh pr create` 실행
* 실제 Pull Request 생성
* 확인되지 않은 변경사항 추측
* 비밀정보 또는 환경변수 값 출력

비밀정보가 변경사항에 포함된 경우 값을 노출하지 않고 다음과 같이 경고한다.

```text
현재 변경사항에 비밀정보가 포함되었을 가능성이 있습니다.
PR 작성 전에 해당 파일의 추적 여부를 확인해야 합니다.
```

---

## 완료 기준

다음 조건을 모두 만족하면 PR 초안 작성을 완료한 것으로 판단한다.

* 실제 변경사항을 확인했다.
* 이번 PR의 목적을 한 문장으로 설명할 수 있다.
* 파일 목록이 아니라 동작 중심으로 변경사항을 정리했다.
* 검증 결과를 실제 실행 여부에 맞게 작성했다.
* 확인되지 않은 내용을 사실처럼 표현하지 않았다.
* 리뷰어가 변경 목적과 중요 판단을 이해할 수 있다.
* 사용자가 GitHub에 복사할 수 있는 제목과 본문을 제공했다.
